Moving & Managing Data with AWS

Three services that all "move data" but do completely different jobs. The whole topic is knowing which job each one is for.

01
Anchor this first

3 services, 3 different jobs

They get confused because all three touch "data going to AWS." Separate them by what job and one-time vs ongoing:

ServiceThe job it doesOne-time or ongoing?How
Storage GatewayOn-prem apps keep working but use AWS as their storage backend.OngoingOnline (local cache)
AWS BackupScheduled, centralized backups + retention across AWS services.OngoingOnline (policy-driven)
Snow FamilyMove a huge one-off pile of data into AWS when the network can't.One-timeOffline (physical shipping)
The one line

Storage Gateway = ongoing hybrid access · AWS Backup = scheduled backups · Snow Family = one-time bulk migration by truck/box.

02
Hybrid bridge

AWS Storage Gateway

A gateway appliance (VM/hardware) sits on-prem, presents the storage protocol your apps already speak, caches hot data locally for low latency, and syncs everything to AWS in the background. Your apps don't know the data went to the cloud.

TypeOn-prem speaks…Stored in AWS as
S3 File GatewayNFS / SMB (files)Objects in S3
FSx File GatewaySMB (Windows shares)Amazon FSx for Windows
Tape GatewayBackup software (virtual tape / VTL)S3 + Glacier (replaces physical tape)
Volume GatewayiSCSI (block volumes)S3 + EBS snapshots

Volume Gateway modes: Cached = primary data in AWS, only hot data cached on-prem · Stored = entire dataset on-prem, async backups to AWS as EBS snapshots.

Trigger

"Keep using the on-premises app but store/back up the data in AWS" → Storage Gateway. Families = File / Tape / Volume.

03
Centralized backups

AWS Backup

One policy-driven place to automate and centralize backups across many AWS services, instead of scripting snapshots per service. Built for retention rules, compliance, and disaster recovery.

  • Backup plan — a policy: how often to back up, retention period, and lifecycle (move old backups to cold storage).
  • Backup vault — the encrypted store where recovery points live.
  • Cross-Region & cross-account copies — for disaster recovery.
  • Backup Vault Lock — WORM (write-once-read-many) immutability; protects against deletion / ransomware and meets compliance.
  • Backup Audit Manager — proves your backup policies meet requirements.
  • Covers EC2/EBS, RDS, Aurora, DynamoDB, EFS, FSx, S3, Storage Gateway volumes, and more.
Trigger

"Centrally manage / automate backups across multiple AWS services" or "meet a backup retention / compliance policy" → AWS Backup (+ Vault Lock for immutable, ransomware-proof backups).

Don't confuse

AWS Backup is for scheduled protection & retention, not bulk one-time migration (that's Snow) and not live hybrid access (that's Storage Gateway).

04
Ship it physically

AWS Snow Family

Physical, rugged, encrypted devices AWS ships to you. You load your data, ship it back, and AWS imports it to S3. For when uploading over the internet would take weeks or months. Sizes: cone < ball < mobile.

DeviceCapacityForm factorBest for
Snowcone~8 TB (HDD) / 14 TB (SSD)Smallest, rugged, fits in a backpackSmall/edge transfers, tight or harsh spaces; light edge compute.
Snowball Edge~80 TB (use several for PB)Suitcase-sizedTB–PB transfers. Storage-optimized for data; compute-optimized (vCPU/GPU) for edge processing.
Snowmobileup to 100 PB45-ft shipping container on a truckExabyte-scale whole-data-center migrations.
  • All devices: encrypted (KMS), tamper-resistant, tracked end to end.
  • Edge computing — Snowcone/Snowball can run EC2 instances or Lambda offline (ships, factories, remote sites) before data is shipped back.
Rule of thumb

If transferring it over your internet connection would take weeks+, use Snow. Size by volume: TB / portable → Snowcone · ~80 TB–PB → Snowball · exabytes / data-center move → Snowmobile.
